# Audience Brief And Intent Lock

Use this before planning the deck. The brief should keep execution focused on
the intended viewer and on the user's actual subject matter.

## Goal

Convert the user's raw request into an audience-centered production brief that
preserves the user's intent, removes ambiguity, and raises the execution bar.

## Brief Shape

```text
User intent:
Intended viewer:
Viewer context:
Viewer need:
Deck job:
Desired takeaway:
Must-cover content:
Tone:
Evidence/source needs:
Visual direction:
Image responsibilities:
Data/chart needs:
Motion expectations:
Output constraints:
Quality bar:
```

## Rules

- Preserve what the user actually asked for. Do not substitute a generic
  explainer, product pitch, or process description.
- Strengthen under-specified parts into concrete production decisions:
  intended viewer, viewer need, deck shape, slide jobs, visual role, image
  roles, chart roles, motion grammar, and QA checks.
- Do not invent facts, metrics, product claims, customer names, sources, or
  business conclusions.
- If the user asks for a final deck, the visible slides should read like a
  finished presentation for the intended viewer: context, argument, evidence,
  visuals, and next step.
- If the user's request conflicts with a skill requirement, satisfy the user's
  content first and use the skill requirement as an internal production method.
- Convert vague visual asks into decisive constraints, but keep user-facing
  choice gates when the root skill requires them.

## Anti-Laziness Bar

Before writing HTML, verify internally:

- The deck answers the user's actual question.
- Each slide has a job and a concrete deliverable.
- The audience can tell why this page matters within 3-5 seconds.
- Images, charts, motion, and layout serve content, not decoration.
- Generated images have explicit responsibilities and are not generic filler.
- No slide is a placeholder, a template demo, or maker-side note.
- The final deck can stand alone for its intended viewer.
